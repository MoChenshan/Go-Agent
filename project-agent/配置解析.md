## 二、本地开发环境

### 2.1 启动命令与原理

```bash
# 快速启动（HTTP 模式）
make run              # → go run . -debug

# CLI 交互模式
make run-cli          # → go run . -cli -debug

# 编译二进制
make build            # → CGO_ENABLED=0 go build -tags "a2a agui iwiki redis" -ldflags "..." -o bin/gameops-agent .
```

**`main.go` 启动时序**（`80:132`）：

```
loadDotEnv()          → 手动读取 .env 文件（docker compose 自动加载，本地 go run 需手动）
flag.Parse()          → 解析 -addr / -model / -cli / -debug 等 flag
config.Load()         → YAML + 环境变量合并，生成 *Config
observability.Init()  → 初始化 OTel TracerProvider / MeterProvider（OTEL_ENABLED 控制开关）
app.Init()            → 构造 Agents / Tools / Plugins / Audit / Session / Cost
runHTTP / runCLI      → 进入对应模式
```

### 2.2 配置加载机制（`src/config/loader.go`）

项目采用**三层优先级**的配置体系：

| 优先级 | 来源 | 说明 |
|---|---|---|
| 1（最高） | 命令行 flag | `-model`、`-addr`、`-debug` |
| 2 | 环境变量 | `MODEL_NAME`、`OPENAI_API_KEY` 等 12-factor 标准做法 |
| 3（最低） | YAML 配置文件 | `-config config.yaml`，不存在则走 `Default()` |

**为什么这么设计**：
- **12-factor 原则**：密钥（API Key）不进代码和 YAML，走环境变量/Secret 注入
- **零配置可启动**：`Default()` 返回合理默认值，`go run .` 即可运行
- **环境变量覆盖 YAML**：同一配置项，环境变量优先级更高，方便 K8s/Docker 注入

### 2.4 Build Tags 机制

```makefile
BUILD_TAGS ?= a2a agui iwiki redis
```

| Tag | 作用 | 原理 |
|---|---|---|
| `a2a` | 启用 A2A 协议（Agent-to-Agent） | 编译时通过 `//go:build a2a` 注入真实实现，否则走 stub |
| `agui` | 启用 AG-UI Web 前端 | 同上，`a.AGUI.Enabled()` 判断是否挂载 `/agui` |
| `iwiki` | 启用 iWiki 知识库 RAG | 编译时注入 iWiki 工具实现 |
| `redis` | 启用 Redis Session/Idempotency | 引入 `go-redis` 依赖和 Redis 适配层 |
| `integration` | 集成测试 | `make test-integration` 使用 |
| `chaos` | 故障注入测试 | `make test-chaos` 使用 |

**为什么这么设计**：外网/离线 CI 无法访问内网依赖（trpc-a2a-go 等），stub 构建保证任何环境都能编译通过。

### 3.3 核心配置详解

#### agent 服务

```yaml
agent:
  build:
    context: .
    dockerfile: Dockerfile
    args:
      VERSION: dev
  depends_on:
    - redis
    - otel-collector     # 必须等 OTel Collector 先启动
  ports:
    - "8080:8080"
  environment:
    # 网络地址用 Docker 内部 DNS（容器名）
    SESSION_REDIS_ADDR: redis:6379        # 不是 localhost！
    OTEL_EXPORTER_OTLP_ENDPOINT: http://otel-collector:4318
    LANGFUSE_HOST: http://langfuse:3000  # 容器间通信用服务名
```

**关键设计**：
- **网络隔离**：容器间用 Docker 内部 DNS（服务名如 `redis`、`otel-collector`），宿主访问用 `localhost:映射端口`
- **环境变量默认值**：`${OPENAI_API_KEY:-sk-stub}` → 优先读宿主 `.env`，缺省用 stub 值
- **健康检查**：`wget -qO- http://localhost:8080/healthz`，10s 间隔，5 次失败后标记 unhealthy

#### Dockerfile 多阶段构建

```dockerfile
# Stage 1: builder (golang:1.24-alpine)
#   - 利用层缓存：先 COPY go.mod go.sum → go mod download
#   - 再 COPY . . → 编译
#   - --mount=type=cache 缓存 /go/pkg/mod 和编译缓存

# Stage 2: runtime (alpine:latest)
#   - 创建非 root 用户
#   - 仅拷贝二进制 + mcp_servers.yaml + skills + guard_rules.yaml
#   - USER nonroot:nonroot（最小权限原则）
#   - ENTRYPOINT ["/app/gameops-agent"]
#   - CMD ["-addr=:8080"]
```

**为什么用多阶段构建**：
1. 编译镜像 1GB+，运行镜像 ~25MB
2. `CGO_ENABLED=0` 编译纯静态二进制，无 libc 依赖
3. `-trimpath -s -w` 去掉调试信息，体积更小
4. `-ldflags "-X main.version=..."` 在编译期注入版本号

#### Redis 配置

```yaml
redis:
  image: redis:7.4-alpine
  command: ["redis-server", "--appendonly", "yes", "--maxmemory", "512mb", "--maxmemory-policy", "allkeys-lru"]
```

- `--appendonly yes`：开启 AOF 持久化
- `--maxmemory 512mb --maxmemory-policy allkeys-lru`：内存上限 512MB，满时 LRU 淘汰
- `redis_data:/data`：数据卷持久化

#### OTel Collector 配置（`deploy/otel/collector.yaml`）

```yaml
receivers:                    # 接收数据
  otlp:                      #   OTLP gRPC(:4317) + HTTP(:4318)

processors:                   # 处理数据
  batch:                     #   批量发送（5s/1024条）
  memory_limiter:            #   内存保护（512MB）
  attributes/redact:         #   PII 哈希脱敏（gen_ai.prompt/completion）

exporters:                    # 输出数据
  otlp/jaeger:               #   Traces → Jaeger
  otlphttp/langfuse:         #   Traces → Langfuse（LLM Token 用量）
  prometheus:                #   Metrics → :8889/metrics 给 Prometheus 抓
  debug:                     #   控制台输出

service:
  pipelines:
    traces: [otlp] → [memory_limiter, attributes/redact, batch] → [otlp/jaeger, otlphttp/langfuse]
    metrics: [otlp] → [memory_limiter, batch] → [prometheus]
    logs:    [otlp] → [batch] → [debug]
```

#### Prometheus 配置（`deploy/prometheus/prometheus.yml`）

```yaml
global:
  scrape_interval: 15s          # 15s 采集一次
  external_labels:
    cluster: gameops-local       # 集群标识

scrape_configs:
  - job_name: otel-collector     # 抓 OTel Collector 暴露的 Prometheus 指标
    static_configs:
      - targets: ['otel-collector:8889']
```

### 4.2 Helm values.yaml 关键配置解读

#### 资源与弹性

```yaml
replicaCount: 3                    # 3 副本起步，保证高可用

resources:
  requests: { cpu: 500m, memory: 512Mi }
  limits:   { cpu: 2000m, memory: 2Gi }

autoscaling:                        # HPA
  minReplicas: 3
  maxReplicas: 20
  targetCPUUtilizationPercentage: 70
  targetMemoryUtilizationPercentage: 80
  customMetrics:                    # 可选：按在飞会话数缩放
    targetSessionsInFlight: 50
```

**为什么这么设**：Agent 是 LLM 密集型，CPU 70% 即触发扩容避免排队；2Gi 内存上限防止单 Pod 吃光节点内存。

#### 稳定性保障

```yaml
podDisruptionBudget:
  minAvailable: 2                   # PDB：至少 2 副本，保证 HITL 不中断

terminationGracePeriodSeconds: 60   # 优雅关闭 60s（含 HITL 等待 + flush）

affinity:
  podAntiAffinity:                  # 反亲和：避免单节点全挂
    preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        topologyKey: kubernetes.io/hostname
```

**`terminationGracePeriodSeconds: 60` 的原理**：
1. Pod 收到 SIGTERM → `preStop` 执行 `sleep 5`（等 Service 端点摘除）
2. `main.go` 中 `srv.Shutdown(ctx 30s)` 拒绝新连接，等 in-flight 完成
3. `a.Shutdown(ctx 20s)` 等 async runner / audit flush
4. OTel Provider Shutdown（在 main defer 中）
5. 总计 5+30+20 = 55s < 60s，留 5s 余量

#### 安全加固

```yaml
podSecurityContext:
  runAsNonRoot: true
  runAsUser: 65532               # 非 root 用户
  fsGroup: 65532

securityContext:
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true    # 只读根文件系统
  capabilities:
    drop: ["ALL"]                 # 丢弃所有 Linux capabilities

networkPolicy:
  enabled: true
  ingressFromNamespaces:
    - ingress-nginx               # 仅允许 Ingress 网关
    - monitoring                   # 仅允许 Prometheus
  allowedEgressCIDRs:
    - 10.0.0.0/8                  # 仅内网出站
  llmEgressDomains:
    - api.openai.com              # LLM API 域名白名单
```

**NetworkPolicy 原理**：
- Ingress：只放行 `ingress-nginx` 和 `monitoring` 命名空间
- Egress：内网 CIDR + 拒绝 `169.254.0.0/16`（**防 SSRF**，阻止 Pod 访问云元数据接口）

#### Secret 管理

```yaml
# K8s Secret（仅示例）
secrets:
  openaiApiKey: ""            # 留空 → 由 ExternalSecrets / Vault 注入
  auditHmacKey: ""

# Helm 模板中：
env:
  - name: OPENAI_API_KEY
    valueFrom:
      secretKeyRef:
        name: {{ include "gameops-agent.fullname" . }}-secrets
        key: openai-api-key
```

**为什么这么设计**：密钥不进 values.yaml（不进 Git），走 K8s Secret + ExternalSecrets Operator 从 Vault 同步。

#### 健康检查

```yaml
livenessProbe:
  httpGet: { path: /healthz, port: http }
  initialDelaySeconds: 10     # 启动 10s 后开始探测
  periodSeconds: 15            # 每 15s 探测
  failureThreshold: 3          # 连续 3 次失败 → 重启

readinessProbe:
  httpGet: { path: /healthz, port: http }
  initialDelaySeconds: 5
  periodSeconds: 5             # 每 5s 探测（更快感知就绪）
  failureThreshold: 3
```

#### ServiceMonitor（Prometheus Operator）

```yaml
serviceMonitor:
  enabled: true
  interval: 15s
  labels:
    release: prometheus         # 匹配 Prometheus Operator 的 serviceMonitorSelector
```

**原理**：Prometheus Operator 自动发现带此 label 的 ServiceMonitor，无需手动配 `scrape_configs`。

## 六、安全防护规则（`deploy/guard_rules.yaml`）

| 规则 | 类型 | 防护目标 |
|---|---|---|
| `prompt_injection_jailbreak` | Input | 越狱指令（忽略系统提示） |

支持**热加载**：`RuleWatcher` 每 5s 轮询文件 mtime，变更后自动替换；解析失败则保留旧规则（fail-safe）。
