# 项目深度技术报告 · 总览篇

> 本系列是基于 `letsgo_server` 真实代码的**三篇深度技术报告**，每一篇都是**可单独讲解 30-45 分钟**的高质量项目素材，覆盖**通用性、技术深度、面试含金量**三重要求。

---

## 📚 三篇定稿目录

### 1️⃣ [项目深度技术报告 · 分布式撮合引擎篇（matchsvr）](./项目深度技术报告-撮合引擎篇.md)

> 一个承载全游戏所有匹配流量的**带算法的微服务**，几乎可以一比一套到外卖派单 / 打车派单 / 广告竞价 / 推荐召回 / 任务调度等任何"撮合类"业务。

**核心技术亮点**：
- **分片无锁 Worker 模型** —— `matchType % N` 让相同类型天然串行
- **SkipList 多维索引 + 双重剪枝** —— O(log N) 范围查询防退化
- **5:1 高低优先级 work-stealing** —— 防饥饿 + 不浪费
- **反射注册 + 三阶段钩子责任链** —— 10 个过滤器零侵入
- **策略工厂 + default 防滚服 + 热开关** —— 8 种匹配策略 + 灰度能力
- **多级超时 + 渐进扩圈** —— 先紧后松，保质兼保量
- **环形桶滑动窗口** —— 无锁、自愈、置信度可控
- **双阶段优雅下线钩子** —— K8s 无损滚动
- **cancelFailMap 时序乒乓法** —— 异步事件乱序修正
- **匹配后优化（两阶段决策）** —— 先可行、再最优

**能回答的面试题**：高并发架构、范围查询、优先级调度、策略模式、责任链扩展、限流熔断统计、K8s 无损、异步时序、长尾隔离、组合优化算法工程化

### 2️⃣ [项目深度技术报告 · 高并发 IM 聊天系统篇（chatsvr）](./项目深度技术报告-IM聊天系统篇.md)

> 一个支撑全游戏 30+ ChatType 的 IM 服务（单聊/组队聊/战局聊/俱乐部/世界频道/社区频道/星球频道等），**底层 70% 架构与微信/QQ/钉钉/飞书的 IM 完全一致**。

**核心技术亮点**：
- **9 子类多态建模 30+ ChatType** —— 模板方法 + 工厂模式教科书级应用
- **DB 原子自增生成群内严格递增 seq** —— 解决分布式时序
- **发送方快照写时冗余** —— 消除历史消息读时 N+1 查询
- **before/after AOP 钩子链路** —— 校验 / 审计 / 扇出集中管理
- **Kona Fiber 协程 per-target 扇出** —— 万人群无阻塞
- **Trace 跨协程显式传递** —— 解决异步链路追踪断链
- **CoLoadingCache 协程化缓存** —— 回源不阻塞 OS 线程
- **batchLoader + 空值占位** —— 防 N+1 + 防缓存穿透
- **按消息类型/频道屏蔽 DB** —— 存储成本可运维
- **软删除 + seq 水位清空** —— 审计保留、多端一致
- **lastReadSeqId 位置语义** —— 幂等、多端合并、实时算
- **本地 / 远程 Service 双实现** —— 进程内零开销、跨进程透明

**能回答的面试题**：IM 架构设计、消息时序、扇出模型、缓存穿透/击穿、写穿透 vs 写回、推拉结合、协程 vs 线程、异步 Trace、多端已读同步、灰度与配置热更新

## 🗺️ 组合使用指南

### 场景一：大厂后端面试（45 分钟技术面）

- 自我介绍：2 分钟
- 项目介绍（挑一篇深讲）：10 分钟
- 技术细节追问（基于所讲项目展开）：15-20 分钟
- 延伸 & 通用题（扩到架构/基建）：10 分钟
- 提问：5 分钟

**推荐讲法**：
- 主讲 **撮合引擎篇**（算法 + 架构硬）或 **IM 聊天系统篇**（通用 + 硬点多）
- 面试官追问到底层时，自然切到 **架构与基建深度篇** 的内容（协程、缓存、服务治理）

### 场景二：系统设计题（Design an IM / Design a Matching System）

直接把对应篇章的**架构总览 + 核心数据结构 + 核心算法**讲出来即可。两篇都自带完整设计图和技术选型表。

### 场景三：简历上的项目描述

建议简历上每篇项目用**3 句话**概括：
1. 项目规模（支撑 X QPS、Y 用户、Z 节点）
2. 我负责的核心模块（匹配算法 / 扇出模型 / 缓存层）
3. 量化收益（延迟降低 X%、成本节省 Y%、吞吐提升 Z 倍）

## 📝 撰写原则（本系列区别于"水账总结"的地方）

1. **看实际代码**：每一个技术点都配**真实代码引用 + 文件路径**，不编造
2. **讲透一个点**：单一亮点至少从"是什么 / 为什么这么做 / 有什么坑 / 通用经验"四个维度展开
3. **给出迁移场景**：所有技术点都映射到业界通用场景，避免"只在本项目有用"
4. **平衡深度与通用**：既有 matchsvr 独有的业务细节，也有所有后端都能用的通用模式
