## 一、项目关键量化数据预估

> 以下数据基于"平均在线 ~30-50万"进行合理预估，面试时可灵活引用。

### 1.1 在线规模与请求量预估

| 指标 | 预估值 | 推算依据 |
|------|--------|---------|
| **日均在线（均值）** | 30-50万 | 业务基准 |
| **日活跃用户（DAU）** | ~100-200万 | 在线/DAU比约1:3~4（游戏行业典型值） |
| **峰值在线** | ~80-120万 | 峰值/均值约2-3倍（晚高峰+周末） |

### 1.2 核心接口QPS预估

> 推算公式：QPS ≈ 在线人数 × 每用户每秒请求频率

| 接口类型 | 单用户频率 | 日常QPS（30万在线） | 峰值QPS（100万在线） | 说明 |
|---------|:---------:|:------------------:|:-------------------:|------|
| **心跳/状态同步** | ~0.2次/s（5s一次） | ~6万 | ~20万 | 长连接心跳+状态上报 |
| **游戏操作请求** | ~0.5-1次/s | ~15-30万 | ~50-100万 | 移动/交互/战斗等CS请求 |
| **匹配请求** | 集中触发 | ~5000-1万 | ~3-5万 | 非持续，排队期集中 |
| **聊天消息** | ~0.05次/s | ~1.5万 | ~5万 | 群聊/私聊/世界频道 |
| **排行榜查询** | ~0.01次/s | ~3000 | ~1万 | 定期拉取/翻页 |
| **支付下单** | 极低频 | ~500-1000 | ~3000-5000 | 非核心高频，但要求高可靠 |
| **配置/资源拉取** | 登录时集中 | ~2000-5000 | ~1-2万 | 登录风暴时集中 |
| **服务间RPC调用** | 扇出放大 | ~100-300万 | ~500万+ | 一次用户请求可扇出3-10次SS调用 |

### 1.3 存储与缓存量化

| 指标 | 预估值 | 说明 |
|------|--------|------|
| **单玩家数据大小** | ~50-120KB | 含角色/背包/社交/活动等模块（超120KB触发告警） |
| **Redis缓存总量** | ~50-100GB | 多节点（MAIN/REGION/MINOR/FARMCRAZY） |
| **Redis操作QPS** | ~50-200万 | 三级缓存L2层，含Pipeline批量 |
| **TcaplusDB操作QPS** | ~5-20万 | L3层持久化，批量读写 |
| **背包格子上限** | 3000 | 硬性资源限制 |
| **监控指标数** | 3000+ | 覆盖业务/系统/JVM/RPC/缓存等 |

### 1.4 面试话术示例

> "我们项目日常在线约几十万，核心游戏操作接口的整体QPS在**十万到数十万**量级，考虑到服务间RPC的扇出放大，内部RPC调用峰值可以达到**百万级**。我们通过**共享内存通信**（微秒级延迟）、**用户态协程**（单线程数万并发）、**三级缓存**（本地→Redis→TcaplusDB）来支撑这个量级。"

### 2.3 Java vs Go 对比（面试加分）

| 维度 | Java | Go | 面试话术 |
|------|------|-----|---------|
| **并发模型** | 线程(Thread) + 虚拟线程(Loom) | goroutine(GMP) | "Java线程是OS线程，Go是用户态协程，我们项目用Kona Fiber实现了类似Go的N:M协程模型" |
| **GC** | G1/ZGC（分代/分区） | 三色标记+写屏障（无分代） | "Java GC更成熟复杂，Go GC追求低延迟" |
| **错误处理** | try-catch异常机制 | error值返回 | "Go的显式错误处理更清晰，Java异常有性能开销" |
| **泛型** | 完善的泛型（类型擦除） | 1.18+引入（类型参数） | "Go泛型还在演进，Java泛型更成熟" |
| **部署** | JVM启动慢、内存占用大 | 编译成二进制、启动快 | "Go适合微服务快速启停，Java适合长期运行的服务" |
| **生态** | Spring全家桶、极其丰富 | 标准库强大、框架轻量 | "Java生态成熟但重，Go轻量但需要造轮子" |
| **适用场景** | 复杂企业级应用、大数据 | 云原生、微服务、中间件 | "K8s/Docker/etcd都是Go写的，说明Go在云原生领域的优势" |

## 三、面试复习优先级清单

### 🔴 P0 — 必会基础（几乎每场面试都会问）

#### 1. 编程语言深度（JD出现率 ~100%）

- **Java方向**: JVM内存模型 → GC原理(G1) → 并发编程(synchronized/AQS/线程池) → 集合框架 → JVM调优 → 线上排障(jstack/Arthas)
- **Go方向**: GMP调度模型 → goroutine/channel → sync包 → GC(三色标记) → 逃逸分析 → pprof性能分析
- **复习时间**: 4-5h
- 项目关联: Kona Fiber用户态协程、ForkJoin并行、G1 GC调优、Arthas/火焰图

#### 2. 数据结构与算法（JD出现率 ~95%）

- 常见数据结构: 数组、链表、哈希表、树（红黑树/B+树）、堆、跳表
- 常见算法: 排序、二分、DFS/BFS、动态规划、贪心、滑动窗口
- 手写题高频: LRU缓存、令牌桶限流、时间轮定时器
- **复习时间**: 3-4h
- 对应文档: [31 算法与数据结构实践](31%20项目中的算法与数据结构实践.md)
- 项目关联: CoLoadingCache(LRU)、RateLimiter(令牌桶)、TimingWheel(时间轮)、DP背包匹配

#### 3. MySQL / 数据库原理（JD出现率 ~80%）

- 索引原理: B+树、聚簇/非聚簇索引、联合索引最左匹配、覆盖索引
- 事务与隔离级别: ACID、MVCC、幻读、间隙锁
- 锁机制: 行锁/表锁/意向锁/死锁检测
- 查询优化: EXPLAIN执行计划、慢查询优化
- TcaplusDB vs MySQL: 全维度对比（项目亮点）
- **复习时间**: 3-4h
- 对应文档: [39 MySQL与数据库原理深度专题](39%20MySQL与数据库原理深度专题.md)

#### 4. Redis 原理与应用（JD出现率 ~80%）

- TCP/IP: 三次握手/四次挥手、粘包拆包、流量控制/拥塞控制

### 大型多人在线游戏服务端 — 后端开发工程师

**技术栈**: Java 11 · Kona Fiber协程 · Protobuf · Redis · TcaplusDB(NoSQL) · Kafka/Pulsar · Kubernetes · Prometheus

**项目概述**: 参与腾讯旗下一款大型多人在线游戏的服务端开发，项目采用 **60+ 微服务** 分布式架构，基于 **共享内存(Tbuspp)** 进行高性能进程间通信，使用 **用户态协程(N:M模型)** 驱动高并发处理。系统日常承载 **数十万在线用户**，核心游戏操作接口 QPS 达 **十万级**，内部 RPC 调用（含扇出）峰值达 **百万级**。

**主要工作**:

- 参与业务功能模块的需求开发与迭代，基于标准化 **五层分层架构**（RPC → Service → Manager → Logic → Data）进行编码，遵循事件驱动解耦模式
- 使用项目 **三级缓存体系**（本地缓存 → Redis → TcaplusDB）进行数据读写，Redis 层 QPS 达 **数十万至百万级**，熟悉 **Cache-Aside** 一致性策略和 **SingleFlight** 防缓存击穿机制的实际应用
- 基于 **Protobuf** 定义和维护服务间通信协议，理解 SS-RPC 的 **9 种路由模式**（一致性哈希、模哈希、元数据路由等）在不同业务场景下的选型
- 使用 **分布式锁**（CacheLockAgent / RedisLock）、**令牌桶限流**、**协程异步调用** 等基础设施工具完成功能开发，理解其实现原理与适用场景
- 参与服务的 **Kubernetes** 容器化部署与灰度发布流程，使用 **Prometheus + Grafana** 监控 **3000+** 指标的服务运行状态，配合分级告警机制定位线上问题
- 通过 **Arthas** 在线诊断和 **async-profiler** 火焰图分析协助排查性能热点，参与 **G1 GC** 参数调优与协程调度优化

**项目亮点**:

- 深入理解 **共享内存通信** 替代传统TCP的高性能RPC设计，进程间通信延迟达 **微秒级**，内部RPC吞吐远超gRPC方案
- 理解项目中 **用户态协程调度模型**（N:M模型），类似Go的goroutine设计理念，单线程支撑 **数万并发**，避免OS线程上下文切换开销
- 熟悉 **多级缓存 + 分布式锁 + 乐观锁** 组合的数据一致性保障方案，理解在大规模在线场景下的CAP权衡与实践取舍
- 了解 **DP背包算法** 在匹配系统中的应用（阵营人数分配优化），以及 **跳表排行榜**、**时间轮定时器**、**令牌桶限流** 等数据结构在工程中的实践
