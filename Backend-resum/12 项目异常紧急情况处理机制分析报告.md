## 2. 熔断机制

### 2.1 战斗用熔断器 - CombatHystrix

**文件位置**: [CombatHystrix.java](/C:/UGit/letsgo_server/WeA/timiutil/src/main/java/com/tencent/timiutil/tool/hystrix/CombatHystrix.java)

**设计原理**：
- **三态模型**：Open（开启）→ Recovering（恢复中）→ Closed（关闭）
- **CPU负载自适应**：根据CPU负载动态调整服务能力
- **渐进式恢复**：通过步进值逐步恢复，避免流量冲击

**核心流程**：

```java
// 状态枚举
public enum Status {
    Open,      // 正常服务
    Closed,    // 熔断关闭（拒绝服务）
    Recovering // 恢复中
}

// 状态切换逻辑
private void procOpen(long currentTimeMillis, double cpuLoad, double avgLoad) {
    if (cpuLoad < targetCpu) {
        setLastValue(rangeEnd);
        return;
    }
    procRecover(currentTimeMillis, cpuLoad, avgLoad);  // CPU过高进入恢复模式
}

private void procRecover(long currentTimeMillis, double cpuLoad, double avgLoad) {
    if (reachBottomLine(currentTimeMillis, cpuLoad)) {  // 超过底线直接熔断
        return;
    }
    if (avgLoad >= targetCpu) {
        setLastValue(Math.max(rangeBegin, lastValue - 2 * step));  // 降低服务能力
    } else if (avgLoad <= targetCpu - safetyRange) {
        setLastValue(Math.min(rangeEnd, lastValue + step));  // 逐步恢复
    }
}
```

**配置参数**：
| 参数 | 含义 | 典型值 |
|------|------|--------|
| `targetCpu` | 目标CPU使用率 | 70% |
| `cpuBottomLine` | CPU使用率底线 | 90% |
| `step` | 调整步长 | 5 |
| `recoverSec` | 恢复检测间隔 | 2秒 |
| `stopSec` | 熔断后停止时间 | 5秒 |

### 2.2 Redis熔断保护 - Break

**文件位置**: [Break.java](/C:/UGit/letsgo_server/WeA/common/src/main/java/com/tencent/cache/Break.java)

**设计原理**：
- **滑动窗口统计**：使用滑动窗口记录成功/失败请求
- **概率性熔断**：根据失败率计算丢弃概率
- **自适应恢复**：随着成功率提升自动恢复服务

**核心算法**：

```java
private boolean accept() {
    BucketCollector bc = new BucketCollector() {};
    this.rw.stat(bc);  // 统计滑动窗口内的数据

    float k = PropertyFileReader.getFloatItem("redis_break_k", K);  // K=1.3
    float acceptWeight = k * bc.count;  // bc.count是成功请求数
    
    // 计算丢弃率，成功数有1.3倍放大，使丢弃率偏保守
    float dropRatio = Math.max(0,
            ((float) (bc.total - protection) - acceptWeight) / (float) (bc.total + 1));

    if (dropRatio <= 0) return true;
    
    // 概率性熔断
    if (RandomGenerator.getInstance().nextDouble() < dropRatio) {
        return false;
    }
    return true;
}
```

### 3.2 令牌桶算法实现

**文件位置**: [RateLimiter.java](/C:/UGit/letsgo_server/WeA/common/src/main/java/com/tencent/util/RateLimiter.java)

```java
/**
 * 令牌桶算法核心逻辑
 * @param maxProduceAmount 桶容量（允许最大突发流量）
 * @param produceRate      令牌产生速率（每秒）
 * @param consumeAmount    本次消费令牌数
 */
private boolean consumeThreadUnsafe(double maxProduceAmount, double produceRate, double consumeAmount) {
    long now = Framework.currentTimeMillis();
    
    if (lastConsumeTimeMs == 0) {  // 首次：填满令牌桶
        currentLeftAmount = maxProduceAmount;
    } else {
        // 计算时间段内产生的令牌
        double incAmount = (now - lastConsumeTimeMs) * produceRate / 1000;
        currentLeftAmount = Math.min(currentLeftAmount + incAmount, maxProduceAmount);
    }
    lastConsumeTimeMs = now;

    if (currentLeftAmount < consumeAmount) {
        return false;  // 令牌不足，限流
    }
    
    currentLeftAmount -= consumeAmount;
    return true;
}
```

### 5.3 缓存兜底策略

```java
// CacheNode 中的DB兜底查询
public <V> CacheResult<V> getCache(String key, ..., Supplier<V> query) {
    try {
        result.val = cmd.get(key);  // 尝试从缓存获取
        
        if (result.val == null && query != null) {
            result.val = query.get();  // 缓存未命中，查询DB
            
            if (result.val == null) {
                this.stat.incrDbFail();
                this.processNotFound(key, type);  // 填充空占位符
            } else {
                setCache(key, result.val, expiry, type);  // 回写缓存
            }
        }
    } catch (BreakException e) {
        result.errCode = CacheErrorCode.BREAK;  // 熔断后返回错误码
    }
}
```

### 9.3 面试高频QA

**Q1: 你们的熔断策略和Netflix Hystrix有什么区别？**
> "Netflix Hystrix是通用的RPC调用熔断，基于固定的错误率阈值和时间窗口。我们的CombatHystrix是**游戏场景定制化**的——它不是按错误率熔断，而是基于CPU负载自适应调整。游戏服务器的核心矛盾是CPU资源，高负载时需要限制新连接而非拒绝所有请求。此外，CombatHystrix的渐进式恢复（按步长逐步恢复处理能力）比Hystrix的半开状态更平滑，避免恢复瞬间的流量冲击。"

**Q2: 异常处理为什么要分三层？NKCheckedException和NKRuntimeException有什么区别？**
> "分层的核心目的是让不同类型的异常有不同的处理策略和日志级别：
> - **NKCheckedException**：外部系统异常（DB超时、RPC失败），属于预期内的可恢复异常，打ERROR日志并监控
> - **NKRuntimeException**：业务逻辑错误（参数校验失败、状态不满足），属于预期内的正常拒绝，打DEBUG日志
> - **Exception**：未预期的程序Bug，打ERROR日志并返回UnknownError
> 这样做的好处是：ERROR日志只包含真正需要关注的问题，不会被大量的业务拒绝淹没。监控告警基于ERROR日志触发，准确率更高。"

**Q3: 功能开关滥用会不会导致代码中到处是if-else？**
> "我们有分层策略避免滥用：
> - **FeatureOpenConf**：运营配置表，通过TypeID查询，不污染业务代码
> - **PropertyFile实时开关**：用于紧急场景的临时开关，上线稳定后删除
> - **灰度开关**：基于uid取模实现，统一封装在`isGrayUser()`方法中
> 关键是每个开关都有生命周期管理——临时开关在功能稳定后必须清理。我们在Code Review中会检查：新增的PropertyFile开关是否设置了清理Deadline。"

**Q4: 如果熔断器误触发（CPU并没有真正过载），怎么办？**
> "CombatHystrix有两个防误触发机制：
> 1. **平均负载检测**：不仅看瞬时CPU，还看`avgLoad`平均值，避免瞬时波动触发熔断
> 2. **安全区间safetyRange**：恢复判断时需要CPU低于`targetCpu - safetyRange`才逐步恢复，避免在阈值附近反复震荡
> 此外，Break熔断器的K=1.3放大系数也是防误触发的设计——成功数被放大1.3倍，使得丢弃率偏保守。即使在高错误率下，仍会通过一部分请求探测服务是否恢复。"
