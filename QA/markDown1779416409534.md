# 📎 题后补注：原题缺答案、易翻车追问的标准答法骨架

> 补注规则：每条都标 **【框架】=trpc-agent-go 给的** / **【自研】=我在 project-agent 里写的**。面试时先答边界，再展开细节。INTERVIEW.md 已答的不重复。

## 补注 0｜先背：框架 vs 自研一句话切分（万能开头）

> 我项目基于 **trpc-agent-go v1.8.1**。框架给了 Agent/Runner/Session/Memory/Knowledge/Tool/Callback/A2A/AG-UI 这套抽象；我在它之上**自研了 6 个模块**：HITL 两段式、HMAC 链式审计、`pkg/resilience` 韧性原语、`src/async` 异步 Runner、`src/idempotency` 幂等键、`eval/judge_tool_selection` 算法 Judge——这些是框架不提供、运维场景必须的。

## 补注 1｜「主子模式下子 Agent 产生幻觉怎么办？」（字节一面 Q5）

| 层 | 措施 | 边界 |
|---|---|---|
| 输入 | Coordinator 拆细任务，子 Agent 看到的 prompt 越具体越不飘 | 【自研】prompt 模板 `src/agents/*/system_prompt.md` |
| 执行 | 强制走 Tool 拿事实，禁止凭记忆答 | 【框架】Function Calling + 【自研】TargetedTool 白名单 `src/tools/targeted.go` |
| 输出 | post-model callback 校验关键字段必须出现在 ToolResult 里 | 【框架】callback hook 点 + 【自研】`src/plugin/output_guard.go` |
| 兜底 | Repair Agent 强制 HITL，幻觉过不了人审 | 【自研】`src/plugin/safety_guard.go` |
| 评测 | LLM-Judge `EvidenceSufficiency` 维度，CI 卡阈值 0.85 | 【自研】`eval/judge_llm.go` + GitLab CI 门禁 |

## 补注 2｜「multi-agent 互相 a2a 停不下来怎么办？」（字节一面 Q4）

- 【框架】`llmagent.WithEndInvocationAfterTransfer(true)`：Coordinator 单轮只允许一次 transfer；
- 【自研 prompt】Coordinator system_prompt 明令"单轮只发起一次 Transfer"——双保险；
- 【自研协议】子 Agent 间不互相 transfer，物理上构造星型拓扑（不是 mesh）；
- 【自研观测】OTel span 加 `agent.transfer.depth`，Prometheus 告警 `depth>3` 直接 page；
- 【可补的话术】"如果做成 mesh，我会加 hop_count + visited_set 做回环检测，类似 BGP loop prevention"。

## 补注 3｜「session 隔离 / 跨项目记忆窜污」（字节二面常问）

- 【框架】`session.Service` 接口允许传 namespace；
- 【自研】Session ID = `tenant_id + user_id + project_id` 三段式 key，底层 Redis 走前缀分区；
- 【自研】Knowledge 向量库 metadata 强制带 `tenant_id`，检索时通过框架的 metadata filter 强制注入，**Agent 看不到别租户的文档**；
- 【自研】Memory namespace 同样三段隔离，A2A transfer 跨 Agent 时 session_id 透传但 tenant 边界由网关层校验。

## 补注 4｜「Agent 执行模式 React/Plan&Execute/Reflection？」（淘天一面）

我项目**三种共存**，按场景选：

| Agent | 模式 | 谁实现的 | 为什么 |
|---|---|---|---|
| Coordinator | **纯路由 Planner**（一次 transfer 就完） | 【框架】`llmagent` + `WithEndInvocationAfterTransfer` | 防发散 |
| DiagnosisAgent | **ReAct** | 【框架】`Planner` 接口 + 【自研】中文化 `src/agents/react.go` | 排障要边查边推 |
| RepairAgent | **Plan→HITL→Execute（两段式）** | **【完全自研】**`safety_guard.go` + SSE `confirmation_required` 事件 | 写操作必须人审 |
| FileAnalystAgent | **ReAct + 沙盒** | 【框架】+ 【自研】路径白名单 | 不让 LLM 越权读文件 |

**没用 Reflection**——理由："运维场景错误代价高、token 预算紧，自反思链路反而增加幻觉机会，我用外部 LLM-Judge + HITL 替代"。

## 补注 5｜「Agent 循环最大次数？防死循环？」（淘天一面）

- 【框架】`llmagent.WithMaxIterations(N)`——ReAct 设 6；
- 【自研】`pkg/resilience/breaker.go` 熔断：连续失败 5 次 open；
- 【自研】`src/cost/cost.go` token 累计阈值，超限直接终止本轮并返回降级文案；
- 【自研】OTel span 数监控，单 invocation > 30 个 tool span 报警。

## 补注 6｜「长记忆 vs 短记忆为什么这样做？为什么偏好不存 MySQL？」（淘天/字节二面）

- 【框架】`session.Service` 装短期 events、`memory.Service` 装长期偏好；
- 【自研决策】偏好需要**模糊召回**（"这用户以前问过哪类问题"），向量比 MySQL LIKE 鲁棒；
- 【自研降级】高频结构化字段（user_id、team、tenant）依然落 MySQL，向量库只存"偏好向量+原文 snippet"；
- 【自研】`src/session/session.go` 配三档总结触发：20 条 / 4k token / 静默 5 分钟，**异步走，不阻塞主链路**，LLM 不可用时降级为纯 events。

## 补注 7｜「a2a 为什么不用 Google ADK 那个协议自己写一套？」（字节二面 Q3）

- 【框架】我**直接用了** `trpc-a2a-go v0.2.5`，没自己造；
- 【架构理由】Google ADK 协议偏向同进程子 Agent 调用，A2A 协议天然支持跨进程跨 Service，**符合公司 tRPC 微服务化部署**；
- 【自研增强】通过 build tag 做了 stub/real 双实现（`src/services/a2a/a2a_stub.go` / `a2a_real.go`），离线 CI 也能编译；
- 【可被追问的弱点】"目前 A2A 跨服务调用没做 mTLS，下一步要补"——主动暴露不足。

## 补注 8｜「shard-state、agent team、主子结构和你这个有什么区别？」（字节二面 Q4）

| 拓扑 | 谁决策 | 状态 | 适用 | 我的选择 |
|---|---|---|---|---|
| **Shared State**（黑板） | 所有 Agent 抢/写 | 中心化 | 协同探索 | ❌ 写竞争难追责 |
| **Agent Team**（mesh） | 平等协商 | 各自维护 + 消息 | 头脑风暴 | ❌ 路由不收敛 |
| **主子（我用的）** | Coordinator 单点决策 | Session 共享 | 流水线式任务 | ✅ 运维场景需要"决策可追溯" |

> 加分话术："运维不是创造性任务，是流程性任务，主子结构 + HITL 才能让审计链可解释。"

## 补注 9｜「MCP schema 格式 / MCP 协议包含什么 / MCP 是自己开发的吗？」（字节/淘天反复问）

- 【框架】`trpc-mcp-go v0.0.14`，遵循 MCP 标准协议（JSON-RPC 2.0 over stdio/HTTP/SSE）；
- 【MCP 内容】3 类原语：**Tools**（可调用函数）、**Resources**（可读数据）、**Prompts**（提示词模板）；schema 走 JSON Schema；
- 【自研】我没自己写 MCP server，用 `mcp_servers.yaml` 声明远端 MCP server 列表（蓝鲸/BCS/工蜂/iWiki），框架自动启动 client；
- 【MCP vs Function Calling 的本质区别】Function Calling 是**模型 API 层**协议，MCP 是**工具进程间**协议。同一个工具，先被 MCP server 暴露，再被 Agent 包装成 Function Calling schema 喂给模型——**两层正交**。

## 补注 10｜「业务 Agent evaluation 流程 / tools 多调对结果无影响怎么打分？」（字节寄面 Q5）

- 【框架】`evaluation` 子包给了基础 Golden Set 框架；
- 【自研】`eval/judge_tool_selection.go` 三个子指标：
 - `precision = 实际调用 ∩ 期望调用 / 实际调用`（无效多调会拉低）；
 - `recall = 实际 ∩ 期望 / 期望`（漏调拉低）；
 - `redundancy_penalty`（连续相同 tool 重复调用扣分）；
- 【自研】`tool_selection_accuracy = f1 × (1 - redundancy_penalty)`，是**算法 Judge，零 LLM 成本**，CI 跑 12 条 golden case，阈值 0.85；
- 【自研】线上对应埋点 `gameops_tool_selection_accuracy` Prometheus metric，离线+在线双通道。

## 补注 11｜「RAG 检索 vs 生成哪一环出问题？」（字节寄面 Q8）

- 【RAG 侧 4 维归因】（每维独立打分，谁低就是谁的锅）：
 1. **Retrieval Recall**：期望文档是否进 top-k？没进 → 召回锅，调 chunk/embedding；
 2. **Rerank Precision**：top-k 里相关文档是否被 rerank 排前？没排前 → reranker 锅；
 3. **Faithfulness**：答案每个事实是否在 context 里？不在 → 生成幻觉锅；
 4. **Answer Relevancy**：答案是否回答了 query？跑题 → prompt 锅；
- 【自研】我用 LLM-Judge 同时打这 4 维，dashboard 拆分展示，**故障定位不靠猜**。

## 补注 12｜「不同文档类别上下文格式不一样怎么处理？」（字节寄面 Q6）

- 【自研】`data/knowledge/` 我按类别分目录（`architecture/` `runbook/` `faq/` `incident/`），框架的 `BuiltinKnowledge` 自动把目录名写进 metadata.category；
- 【自研】不同 category 走不同 chunker：runbook 按"步骤编号"切（语义边界），架构文档按 markdown header 切，告警 incident 按"时间窗口"切；
- 【自研】query 时 LLM 先判类型，metadata filter 注入到检索（`filter: {category: "runbook"}`），**避免 runbook 和 FAQ 互相污染**；
- 【翻车修复话术】"如果当时面试这道题，我应该说：'按 category 分 chunker、metadata filter 路由'，不应该说'预处理默认一样'"。

## 补注 13｜「向量数据库怎么选索引？怎么降低 RAG latency？」（淘天三面 Q39/Q40）

- 【框架】框架抽象 `VectorStore` 接口，已支持 InMemory / pgvector / TcVector / Milvus；
- 【自研选型】
 - 文档 < 10 万：InMemory + 暴力余弦，零依赖；
 - 10 万 ~ 1000 万：pgvector + HNSW（M=16, ef_construction=200），单机够用；
 - > 1000 万：Milvus IVF_PQ 分片，损一点 recall 换 10x QPS；
- 【自研降 latency】
 - 一级缓存：query embedding LRU，命中率 ~30%；
 - 二级缓存：top-k 文档片段 Redis 缓存，TTL 5min；
 - 并行召回：dense + sparse 走 errgroup 并行，wall time 砍半；
 - 提前过滤：metadata filter 推下到向量库层，**先过滤再算距离**。

## 补注 14｜「百万用户 Agent 平台怎么设计？」（淘天三面 Q25）

- **接入层**：Ingress + Gateway，按 user_id 一致性哈希到 Pod，保证 session 粘性；
- **执行层**：Pod HPA 1-200，每 Pod 跑 N 个 worker；
- **状态层**：Session 走 Redis Cluster，分片 key=user_id；
- **【自研】异步降级**：长任务（> 30s）自动转 `src/async/runner.go` 异步 Job，前端轮询 `job_status`，**不占连接池**；
- **【自研】成本闸**：`src/cost/cost.go` 按 tenant 配 token 预算，超限直接 429；
- **模型路由**：cheap model（DeepSeek-V3）兜 80% 简单意图，expensive model（Claude）只兜复杂排障；
- **观测**：每个 user 的 invocation 走独立 trace_id，Langfuse session 视图一键回溯。

## 补注 15｜「checkpoint / resume 怎么做？」（淘天三面 Q24）

- 【框架】Session 的 events 本身就是 append-only log，重启后从 events 重放即可；
- 【自研】异步 Job 的 state machine 持久化在 `src/async/store.go`：`pending→running→{succeeded/failed/cancelled/timed_out}`，**进程崩溃后 watchdog 把 running 超时的 Job 转 failed**；
- 【自研】HITL 中断态写 `pending_approval`，前端 approve 后从断点续跑——这是 **流式 SSE 的最大难点**，状态对齐通过 invocation_id + step_id 双键定位。

## 补注 16｜「如何让上下文一致 / 5 天开发不目标偏移？」（字节二面）

- 【框架】Session 自动总结：每轮事件压缩成 summary 注入下轮 prompt；
- 【自研】**Goal Pinning**：把"本次任务目标"写进 session metadata 而非 events，每轮 prompt 头部强制注入，**不会被滚动窗口刷掉**；
- 【自研】重要决策落 `src/audit/hmac.go`（不可篡改），HITL 确认记录可回放；
- 【自研】evalrun 跑 golden 防漂移，CI 阻断退化 MR。

## 补注 17｜「云端版本 / 多租 / 逻辑多租怎么做？」（字节二面）

- **物理隔离层**：每个租户独立 namespace，K8s NetworkPolicy 隔离；
- **逻辑隔离层**：
 - Session/Memory key 三段式（已答）；
 - Knowledge 向量库 metadata.tenant_id 强制 filter；
 - 工具白名单按租户配（`tools.FilterByTargets` 加 tenant 维度）；
- **资源配额**：`src/cost/cost.go` 改造支持按 tenant 限 token / QPS；
- **审计**：HMAC kid 按租户分发，跨租户篡改不可能。

## 补注 18｜「常见 LLM 八股漏题（你面经里没背过的）」

| 题 | 一句话答 |
|---|---|
| KV Cache 是什么？为什么慢？ | 注意力算 K/V 复用上轮缓存；context 越长 KV 占显存越大，**PagedAttention 用页式管理解决 OOM** |
| 为什么用 RoPE | 旋转位置编码，相对位置 + 长度外推性好 |
| 为什么用 RMSNorm 不用 LayerNorm | 少算一个均值，速度快；LLaMA 系列已验证效果不掉 |
| Speculative Decoding 是什么 | 小模型猜 N 个 token，大模型并行验证，2-3x 推理加速 |
| Function Calling 在模型侧怎么训出来的 | SFT 阶段构造 `<tool_call>{...}</tool_call>` 格式样本 + RLHF 偏好对齐 |
| Reflection / Reflexion 区别 | Reflection=单次自我批评；Reflexion=带"经验池"跨 episode 学习，本质 RL |
| Tree-of-Thoughts | 把 CoT 推理树展开，BFS/DFS + 自评 prune，但**成本指数级**，运维场景不用 |
